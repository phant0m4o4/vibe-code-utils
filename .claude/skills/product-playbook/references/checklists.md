# 各阶段详细清单

进入某个阶段时只读对应小节。每节是该阶段要逐条过的问题/检查项,产物文件见 SKILL.md 总览表。

## 阶段 1:概念提出

逐条回答:

- 用户有什么痛点?痛点是否真实存在?
- 高频还是低频?刚需还是伪需求?
- 这个问题现在用户怎么解决?为什么现有方案不好?
- 我们凭什么能做得更好?

不要急着画页面、列功能。产出就是那句填空:为【什么用户】,在【什么场景】下,解决【什么问题】,通过【什么产品能力】,带来【什么价值】。

## 阶段 2:用户研究

四种方法,按可得性选用:

**定性访谈**,要问:你现在怎么解决?最痛苦的是什么?为什么不用现有产品?什么情况下会用新产品?愿意付费吗?最不能接受什么?
注意:用户说"我会用"不一定真。他说"想健康饮食"但天天点外卖 → 研究他为什么做不到,而不是信他说的话。

**定量调研**(问卷):多少人有这个问题?多高频?付费意愿?人群差异?哪类用户最强烈?

**已有产品数据**:哪个环节流失?哪些功能使用率高?搜索词?反馈和客服问题集中在哪?

**竞品评论分析**:App Store / 小红书 / 知乎 / 微博 / 社群,**重点看差评**——差评暴露真实机会。

## 阶段 3:需求定义

每条需求拆三层:

- **用户需求**:用户真正想完成什么(不是功能名)。
- **产品需求**:产品要提供什么能力。
- **业务需求**:对业务的帮助(日活/留存/转化/数据资产/付费基础)。

写法:场景 → 问题 → 目标 → 方案 → 指标。禁止写成功能罗列。

每条需求标注证据来源(访谈编号/数据/竞品差评)。没有证据但值得探索的想法,单列「假设待验证」清单并标【假设】,不混入正式需求池。

"不编造"不等于"留白":某一层(尤其业务需求层)信息不足时,写出标【假设】的最佳推断 + 对应的确认问题,不要只放"待补"占位——占位符没有给用户任何可以点头或纠偏的东西。

## 阶段 4:目标设定

**业务目标**示例:首月新增 10 万、7 日留存 25%、付费转化 3%、日活 +15%、客服咨询 -30%。
**用户行为目标**示例:完成一次核心动作、收藏、次日再打开。
行为指标优先盯——嘴上说喜欢没用,行为才是真的。

## 阶段 5:MVP 设计

- 列出脑中全部功能 → 做减法,砍到只剩唯一核心闭环。
- 典型该砍项:社区、商城、积分、会员、直播、硬件、共享、大数据。
- 验收问句:"用户能不能用它完成 ___?"
- MVP 不是粗制滥造:功能少,但闭环必须完整可走通。

## 阶段 6:产品方案

四块都要有:

**用户流程**:从进入到完成目标的每一步,越短越好——每多一步流失一批人。
**功能结构**:拆模块。
**信息架构**:什么进首页、什么进二级页、什么高频入口、什么隐藏。用户量大之后任何入口调整影响都巨大。
**规则与异常**(重点):没填必填项怎么办?生成/请求失败怎么办?数据缺失怎么办?网络断了怎么办?内容不安全怎么办?未成年人怎么办?

## 阶段 7:PRD

11 个必备小节:

1. 项目背景(为什么做)
2. 项目目标(指标)
3. 用户场景
4. 功能范围(本期做什么、**不做什么**)
5. 用户流程图
6. 页面说明(内容、按钮、状态)
7. 交互规则(点击后发生什么)
8. 业务规则(推荐/权限/状态流转/计费)
9. 异常状态(空数据、失败、加载中、无权限、网络错误)
10. 埋点需求(点击、成功、失败、收藏、开始、完成、退出、评价)
11. 验收标准(做到什么程度算完成)

标准:所有人看完不会误解,不是越长越好。

## 阶段 8:UX / UI

**原型分层(先便宜后贵,别一上来抠像素)**:

1. **低保真线框**:先用线框/草图定**流程和信息层级**——哪些进首页、跳转顺不顺、每步要不要。改流程在这步改,最便宜。
2. **高保真视觉**:线框定稿后再做视觉(色彩/字体/组件),别在流程没定时就抠像素。
3. **可交互原型**:把页面连起来能点,验证真实跳转与手感。

**先画关键流程**:核心闭环 + 高频路径优先;**异常态必须画**——空/失败/加载中/无权限/断网/不安全内容(呼应"异常流程 > 正常流程",只画正常态的设计不过关)。原型里的页面要和 PRD 的页面说明一一对得上。

**可用性走查(便宜又高价值,别跳)**:拿原型找几个目标用户各走一遍核心任务,只看"卡在哪、哪步要问人"——早期 5 个人就能暴露大半问题。看行为,不要只问"好不好看"。

**UX 看**:入口是否明显、操作是否简单、文案是否易懂、用户是否知道下一步、失败有无补救、新用户能否快速上手。
**UI 看**:品牌调性、色彩、字体、图标、组件一致性、动效、暗色模式、多端适配。
**设计交付**:标注 / 切图 / 组件规范交给研发;规模大的产品建设计系统(统一组件规范),否则功能一多就乱。

## 阶段 9:技术评审

不是问一句"能不能做",六项都要过:

1. **系统架构**:前端/后端/数据库/缓存/队列/对象存储/AI 服务/第三方接口怎么配合。
2. **接口设计**:请求参数、返回字段、错误码、权限校验、超时处理。
3. **性能**:首屏时间、生成等待时间、并发、峰值 QPS、数据库压力、缓存策略。
4. **安全**:登录鉴权、数据加密、防刷、风控、敏感内容过滤、隐私保护。
5. **可扩展性**:不过度设计,也不写死。
6. **成本**:AI 项目必算 token / 模型调用 / 图片生成 / 语音识别 / 存储成本。

## 阶段 10:排期立项

角色:产品、UI、前端、后端、算法、测试、运维、数据分析、运营、法务/合规——按规模取舍。
阶段拆解:需求 → 设计 → 研发 → 测试 → 上线 → 运营,每段有明确出口。
**必须留缓冲**,真实项目会遇到:需求变更、联调问题、第三方异常、测试 bug、审核延迟、埋点错误、线上兼容问题。

## 阶段 11:研发跟进(产品视角)

PRD 写完不是结束:

- **需求答疑**:空态显示吗?未登录能用吗?失败扣不扣次数?记录存多久?删了能恢复吗?——快速决策,别让研发卡住。
- **变更管理**:中途改需求必须评估影响(排期/测试/接口/设计/上线目标),禁止"顺便加一下"。
- **每日同步**:什么完成了、什么延期了、什么有风险、什么要协调。
- **体验走查**:提前试用开发版本,别等测试阶段才发现方向不对。

## 阶段 12:测试验收

七类测试:

1. 功能(对照 PRD 逐项)
2. 流程(完整路径走通)
3. 异常:无网/弱网/超时/空数据/权限不足/登录过期/生成失败/重复提交/支付失败
4. 兼容:系统、机型、浏览器
5. 性能:高并发、响应时间、数据库压力
6. 安全:防注入、防刷、防越权、敏感信息泄露
7. **埋点**:数据是否正确上报——最容易被忽略,埋点错了上线等于盲跑

## 阶段 13:上线准备与灰度

**上线清单**:功能完成、测试通过、埋点验证、后台配置、客服话术、运营素材、监控开启、回滚方案明确。

**灰度放量**:内部 → 1% → 5% → 10% → 30% → 50% → 100%。每步看崩溃率、投诉率、接口错误率,异常即停。

**回滚方案**:怎么关入口?怎么回退版本?数据可恢复吗?有没有远程开关(不发版就能关功能)?影响老用户吗?

## 阶段 14:数据验证

六类数据,按漏斗看:

1. **曝光**:低 → 可能是入口问题
2. **点击**:低 → 文案/入口/认知问题
3. **转化**:核心行为每一步的转化率
4. **留存**:次日/3 日/7 日,留存比新增重要
5. **收入**(如有):付费转化率、客单价、续费率、退款率、LTV(用户生命周期总价值)、CAC(获客成本)
6. **质量**:崩溃率、接口错误率、加载时间、生成失败率、投诉率、客服量

结论必须回答:**是否达到阶段 4 设定的目标?**

## 阶段 15:发布与运营

- **冷启动**:种子用户来源——老用户导流、社群、KOL、内容平台、私域、广告、线下、合作伙伴。
- **用户教育**:新手引导、教程、示例、案例、FAQ、短视频演示。
- **活动**:首次奖励、邀请、打卡、会员试用、限时权益、榜单。
- **内容运营**(AI/社区/工具类尤其重要):模板、案例库、热门内容、用户故事、技巧。
- **反馈收集**:应用内反馈、客服、社群、评论区、问卷、访谈、数据异常分析。

## 阶段 16:迭代优化

优先级五因子:影响用户规模、影响业务目标、问题严重程度(瑕疵还是阻断)、开发成本、战略价值。
公式:优先级 = 用户影响 × 业务价值 × 紧急程度 ÷ 实现成本。
迭代类型:体验优化、功能增强、商业化、增长功能、平台能力。

## 阶段 17:复盘沉淀

四问:

1. 目标达成了吗?(原定 vs 实际 vs 差距)
2. 哪里做对了?(场景判断准 / MVP 控制好 / 灰度有效 / 冷启动成功)
3. 哪里做错了?(需求不清 / 成本低估 / 埋点遗漏 / 入口太深 / 教育不足 / 节奏过快)
4. 下一步:继续投入、调整方向,还是停止?

复盘不是追责。不是所有项目都该继续——数据证明用户不需要就及时止损。
