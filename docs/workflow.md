# 从点子到产品的工作流

把一个点子做成完整软件产品,可以拆成 7 个阶段。核心原则是:**越靠前的阶段越便宜,越靠后越贵**——所以要在便宜的阶段尽可能多地消灭不确定性。

每个阶段都要**落地一个可检查的产物**(一份文档或一段能跑的代码),否则进度无从跟踪、换人无法接力。下面每一步都标了它的产物落到哪个文件;完整的文件体系与进度跟踪规则见文末[「产物清单与进度跟踪」](#产物清单与进度跟踪)。

> 下文提到的 `docs/requirements.md`、`PLAN.md` 等,指的是**你正在做的那个产品项目**里的文件,不是本仓库的文件。

## 1. 锁需求(先想清楚"为谁解决什么问题")

这是第一要务,也是最容易被跳过、代价最大的一步。在写任何代码、画任何图之前先回答:

- **谁用**:目标用户是谁,他们现在怎么解决这个问题?
- **解决什么痛点**:不是"我想做个 X",而是"用户因为没有 X 而具体痛在哪"。
- **凭什么是你**:已有的替代方案哪里不够好?
- **形态**:Web / 移动 App / CLI / 桌面 / 浏览器插件 / API 服务?形态决定后面一切技术选型。
- **规模边界**:是给自己用、给 10 个人用,还是要扛百万用户?早期千万别按百万级去设计。

> 自用脚本别被这串问题绊住:「谁用 / 凭什么是你」答案就是你自己,一句带过;重点想清**形态**和**规模**。

**产物**:`docs/requirements.md` —— 一两段话讲清楚的"问题陈述" + 形态 + 规模边界。**讲不清楚就别往下走。**

## 2. 定义 MVP(砍到不能再砍)

把脑子里那一堆功能列出来,然后做减法:

- 找到**唯一的核心闭环**——用户能完整走完一次、并从中得到价值的最短路径。
- 其余功能全部标记为"以后再说",不进 MVP。
- 用一句话验收:"用户能不能用它完成 \_\_\_?" 能,就是合格的 MVP 边界。

**产物**:`docs/requirements.md` 里的「MVP 范围」小节 —— 核心闭环一句话 + 砍掉/延后的功能清单。

## 3. 设计(技术 + 体验,各一份)

- **技术设计**:技术栈、数据模型、关键模块划分、对外接口。规模小就别上微服务、别提前优化。
- **体验设计**:关键页面/流程的草图或线框图(不用精细)。哪怕是 CLI,也要先把命令和输出长什么样定下来。

**产物**:`docs/architecture.md`(技术设计) + `docs/ux.md`(体验设计;页面线框**或 CLI 命令/输出**,纯后端/库可省) —— 能让另一个人照着动手的方案。

## 4. 搭骨架(让"空壳"先能跑起来)

- 初始化仓库、项目结构、依赖、最基本的 CI。
- 跑通一条"端到端的空管道":前端能调到后端,后端能连上数据库,哪怕只返回 hello。
- 这一步的目的是**消除环境和集成的不确定性**,而不是实现功能。

**产物**:能跑起来的仓库骨架(代码本身) + `PLAN.md` 初始化(记下当前阶段、已完成产物、下一步)。

## 5. 迭代实现(一次一个垂直切片)

- 按"垂直切片"而不是"水平分层"来做:每次交付一个**用户能感知的完整小功能**,而不是"先把所有数据库表建完"。
- 每个切片配测试,边写边测。
- 频繁提交,保持主干随时可运行。
- **一次只做一个切片**,做完停下等开发者确认,再开下一个——别连着把多个切片一口气做完。

**产物**:每个切片 = 一组代码 + 测试 + 一次提交;每完成一个切片就更新 `PLAN.md`(挪动 Current Step、追加 Changelog)。

## 6. 验证 + 加固

严格程度随"是否对外"走:**自用**项目功能验证够了;**一旦对外**(多人用 / 公网可达 / 碰真实用户数据),安全和边界就是硬门槛,不能省。

- **功能验证**(都要):真的去跑、去点,而不只是看测试绿。
- **边界与异常**(对外必做):空输入、超大输入、网络失败、并发。
- **安全**(对外必做):认证授权、输入校验、密钥不进代码库。
- **性能**:只在真有瓶颈时优化,别凭感觉。

**产物**:验证清单及结果(记到 `PLAN.md` 或单独 `docs/verification.md`)—— 逐项标注"已验证/有问题",失败项进技术债。

## 7. 发布 + 收集反馈(然后回到第 1 步)

- 部署、上线、写最基础的文档(怎么装、怎么用)。
- 把它交到真实用户手里,收集反馈。
- 反馈会修正你对第 1 步"需求"的理解——**这是一个循环,不是一条直线。**

**产物**:`docs/usage.md`(怎么装、怎么用) + 反馈记录(回灌到 `docs/requirements.md` 启动下一轮)。

## 产物清单与进度跟踪

每个阶段的产物落到固定文件,合起来就是一套"换人能接力"的项目文档。**文件是唯一权威(SoT,Single Source of Truth,单一来源),`PLAN.md` 只跟踪进度、不重复内容。**

### 文件体系

```
project-root/
├── AGENTS.md              # 治理:AI 代理的行为约束(贯穿全程,见本仓库 AGENTS.md)
├── PLAN.md                # 进度跟踪:当前阶段 / Changelog / 下一步 / 技术债
├── docs/
│   ├── requirements.md    # 阶段 1-2:问题陈述 + 形态 + 规模 + MVP 范围
│   ├── architecture.md    # 阶段 3:技术栈 / 数据模型 / 模块 / 接口
│   ├── ux.md              # 阶段 3:体验设计(页面线框 / CLI 命令输出;纯后端/库可省)
│   ├── verification.md    # 阶段 6:验证清单结果(可单独成文件,也可并入 PLAN.md)
│   ├── usage.md           # 阶段 7:怎么装、怎么用
│   └── adr/               # 可选:不可逆决策的记录(YYYY-MM-DD-NN-标题.md)
└── <你的代码 / 测试 / 配置>
```

### 按规模裁剪(不是每个项目都要全套)

上面是"团队/对外产品"的完整清单。**规模越小,砍得越狠**——给一个 50 行自用脚本套全套文档纯属负担。照阶段 1 定的规模对号入座:

| 规模 | 留哪些产物 | 流程怎么压 |
| --- | --- | --- |
| **自用脚本 / 一次性** | 代码 + 顶部几行说明(干嘛的、怎么跑)足矣 | 7 阶段压成"想清楚 → 写 → 自己跑一遍",不建 docs |
| **小工具(给少数人用)** | `requirements.md`(几行即可) + `PLAN.md` + 基本 `usage.md` | 设计在脑子里过一遍,architecture 可省 |
| **单人产品** | 上一档 + `architecture.md` + 视情况 `verification.md` | ux 纯后端可省;adr 按需 |
| **团队 / 对外产品** | 全套(含 `ux.md` / `verification.md` / `AGENTS.md` / `adr/`) | 一步不省 |

例外:**只要对外**(公网可达 / 多人用 / 碰真实用户数据),哪怕团队很小,阶段 6 的安全与验证也按高档走。**拿不准规模就问开发者,别自己拍。**

### 阶段 → 产物映射

| 阶段     | 产物                   | 落到哪                                     |
| -------- | ---------------------- | ------------------------------------------ |
| 1 锁需求 | 问题陈述 + 形态 + 规模 | `docs/requirements.md`                     |
| 2 MVP    | 核心闭环 + 砍掉清单    | `docs/requirements.md`(MVP 范围小节)       |
| 3 设计   | 技术方案 / 体验草图    | `docs/architecture.md` / `docs/ux.md`      |
| 4 搭骨架 | 能跑的空管道           | 代码骨架 + `PLAN.md` 初始化                |
| 5 迭代   | 垂直切片               | 代码 + 测试 + 提交,更新 `PLAN.md`          |
| 6 验证   | 验证清单结果           | `PLAN.md` 或 `docs/verification.md`        |
| 7 发布   | 使用文档 + 反馈        | `docs/usage.md`,反馈回灌 `requirements.md` |

### PLAN.md 怎么写

跟踪进度只需要四块,别写成第二份设计文档:

- **Current Step**:现在在第几阶段、具体在做什么。
- **Changelog**:已完成的产物/切片,倒序追加,一行一条。
- **Next Step**:下一步要做什么。
- **技术债 / TBD**:已知但暂缓的问题、待开发者拍板的 `[假设]`。

新会话或新人接手时,读 `AGENTS.md` → `PLAN.md` 的 Current Step → 关联产物文档,几分钟内对齐进度,不用从头复述。

> `PLAN.md` 阶段 4 才随仓库建起来。在那之前(阶段 1-3),把当前进度和待确认点先记在 `docs/requirements.md` 顶部,别让前三步的状态只活在对话里。

## 接入已有项目(不是从零开始时)

上面 7 阶段默认"绿地新建"。如果你是接手一个已有代码库加功能,流程这样映射:

- **阶段 1-3 改成"逆向补档"**:不是凭空设计,而是**照现状**把 requirements / architecture 补出来——先读代码、把它跑起来,记下"它现在到底是什么、怎么搭的",未知处标 `TBD` 问开发者。产物是"现状快照",不是"理想方案",别借机重写。
- **阶段 4 搭骨架通常跳过**:骨架已存在,只补缺的部分(比如没 CI 就加最小 CI),跑通一次确认环境能用即可。
- **从阶段 5 正式切入**:你要加的功能就是第一个垂直切片,照"一次一个切片、做完停下确认"推进。
- **产物按规模裁剪同样适用**:别给一个已有小工具硬塞全套文档,缺什么补什么。

一句话:接入 = 先用阶段 1-3 把"现状"摸清落档,再从阶段 5 干活。

## 贯穿全程的三条原则

1. **前轻后重**:需求和设计阶段多花时间,远比写完代码再返工便宜。
2. **持续可运行**:任何时刻主干都能跑,避免"攒一大坨最后集成爆炸"。
3. **小步快验**:每一步都产出可验证的东西,别等到最后才发现方向错了。

## 和开发者的交互

如果这套流程由 AI 助手来驱动,那么"怎么和开发者协作"和"做什么"同等重要——它决定了产出的是你真正要的东西,还是 AI 自己脑补出来的东西。贯穿全程的交互规则:

### 一次只走一步,走完停下等确认

这是最根本的节奏约束,优先级高于下面所有规则:

- **默认逐步**:走完一个阶段(阶段 5 则是走完**一个**垂直切片)就**停下来**,摆出"这步做了什么 / 产物在哪 / 要你确认或拍板什么",拿到开发者明确"继续"才进下一步。
- **绝不攒着一次推一大坨**:一口气把好几个阶段、好几个切片做完,开发者根本审不过来——没被发现的偏差会一路带到更贵的后面阶段才爆。小步确认才来得及纠偏。
- **确认的是"成果",不是每个动作**:阶段内的可逆细节(写代码、跑测试、本地 commit)不用逐个问;但**每产出一个完整产物 / 一个切片**,就是一个必须停下的确认点。

### 不瞎补默认值

- 关键信息没明示时,**停下来问**,而不是用"行业通识"自动填(JWT、80% 覆盖率、默认目录结构、典型字段……都不许擅自补)。
- 实在要先往下走,就标记成 `TBD` 或 `> [假设] ...`,集中列出来等开发者逐条审查,绝不偷偷塞进代码。

### 每个阶段都有确认 gate

| 阶段              | 开发者做什么                            | AI 做什么                                     |
| ----------------- | --------------------------------------- | --------------------------------------------- |
| 1 锁需求          | 回答定位 / 形态 / 规模 / 是否对外等问题 | 提问、复述理解,确认后才往下走                 |
| 2 MVP             | 拍板核心闭环、砍掉哪些功能              | 列出候选范围 + 建议的取舍,等定夺              |
| 3 设计            | 审查技术 / 体验方案,挑选权衡            | 给出方案和取舍,**不替开发者做不可逆决策**     |
| 4-5 搭骨架 / 迭代 | 逐切片确认、给业务方向、回答求助        | 一次只做一个切片:写码/测试/本地提交自动做,**每个切片做完停下等确认再开下一个** |
| 6-7 验证 / 发布   | 验收、决定上线时机                      | 报告真实结果(失败就说失败),不替开发者点"发布" |

### 把决策权和执行权分开

- **可逆的执行**(写代码、跑测试、本地提交)→ AI 自动做,事后能 review。
- **不可逆的动作**(`git push`、部署、删数据、对外发请求)→ **永远手动确认**,AI 不擅自触发。

### 卡住不死磕

- 同一个点连续卡几次(比如 3 次)就**升级成"求助"**:停下来,讲清楚卡在哪、试过什么、需要开发者什么决定,而不是反复试错烧时间/token。

### 状态可接力

- 进度、决策、当前在做哪一步都落到文档里(`PLAN.md` + 各产物文件),而不是只存在某次对话里,换个新会话或新人也能读几个文件就跟上,不用从头复述。

> 一句话:**AI 负责把活干快干对,开发者负责掌舵和按下不可逆的按钮。** 边界越清楚,协作越顺。
