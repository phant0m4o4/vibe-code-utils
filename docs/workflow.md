# 从点子到产品的工作流

把一个点子做成完整软件产品,可以拆成 7 个阶段。核心原则是:**越靠前的阶段越便宜,越靠后越贵**——所以要在便宜的阶段尽可能多地消灭不确定性。

## 1. 锁需求(先想清楚"为谁解决什么问题")

这是第一要务,也是最容易被跳过、代价最大的一步。在写任何代码、画任何图之前先回答:

- **谁用**:目标用户是谁,他们现在怎么解决这个问题?
- **解决什么痛点**:不是"我想做个 X",而是"用户因为没有 X 而具体痛在哪"。
- **凭什么是你**:已有的替代方案哪里不够好?
- **形态**:Web / 移动 App / CLI / 桌面 / 浏览器插件 / API 服务?形态决定后面一切技术选型。
- **规模边界**:是给自己用、给 10 个人用,还是要扛百万用户?早期千万别按百万级去设计。

产出物:一两段话能讲清楚的"问题陈述"。**讲不清楚就别往下走。**

## 2. 定义 MVP(砍到不能再砍)

把脑子里那一堆功能列出来,然后做减法:

- 找到**唯一的核心闭环**——用户能完整走完一次、并从中得到价值的最短路径。
- 其余功能全部标记为"以后再说",不进 MVP。
- 用一句话验收:"用户能不能用它完成 ___?" 能,就是合格的 MVP 边界。

## 3. 设计(技术 + 体验,各一份)

- **技术设计**:技术栈、数据模型、关键模块划分、对外接口。规模小就别上微服务、别提前优化。
- **体验设计**:关键页面/流程的草图或线框图(不用精细)。哪怕是 CLI,也要先把命令和输出长什么样定下来。

产出物:一份能让另一个人照着动手的方案。

## 4. 搭骨架(让"空壳"先能跑起来)

- 初始化仓库、项目结构、依赖、最基本的 CI。
- 跑通一条"端到端的空管道":前端能调到后端,后端能连上数据库,哪怕只返回 hello。
- 这一步的目的是**消除环境和集成的不确定性**,而不是实现功能。

## 5. 迭代实现(一次一个垂直切片)

- 按"垂直切片"而不是"水平分层"来做:每次交付一个**用户能感知的完整小功能**,而不是"先把所有数据库表建完"。
- 每个切片配测试,边写边测。
- 频繁提交,保持主干随时可运行。

## 6. 验证 + 加固

- **功能验证**:真的去跑、去点,而不只是看测试绿。
- **边界与异常**:空输入、超大输入、网络失败、并发。
- **安全**:认证授权、输入校验、密钥不进代码库。
- **性能**:只在真有瓶颈时优化,别凭感觉。

## 7. 发布 + 收集反馈(然后回到第 1 步)

- 部署、上线、写最基础的文档(怎么装、怎么用)。
- 把它交到真实用户手里,收集反馈。
- 反馈会修正你对第 1 步"需求"的理解——**这是一个循环,不是一条直线。**

---

## 贯穿全程的三条原则

1. **前轻后重**:需求和设计阶段多花时间,远比写完代码再返工便宜。
2. **持续可运行**:任何时刻主干都能跑,避免"攒一大坨最后集成爆炸"。
3. **小步快验**:每一步都产出可验证的东西,别等到最后才发现方向错了。

## 和开发者的交互

如果这套流程由 AI 助手来驱动,那么"怎么和开发者协作"和"做什么"同等重要——它决定了产出的是你真正要的东西,还是 AI 自己脑补出来的东西。贯穿全程的交互规则:

### 不瞎补默认值

- 关键信息没明示时,**停下来问**,而不是用"行业通识"自动填(JWT、80% 覆盖率、默认目录结构、典型字段……都不许擅自补)。
- 实在要先往下走,就标记成 `TBD` 或 `> [假设] ...`,集中列出来等开发者逐条审查,绝不偷偷塞进代码。

### 每个阶段都有确认 gate

| 阶段 | 开发者做什么 | AI 做什么 |
|---|---|---|
| 1 锁需求 | 回答定位 / 形态 / 规模 / 是否对外等问题 | 提问、复述理解,确认后才往下走 |
| 2 MVP | 拍板核心闭环、砍掉哪些功能 | 列出候选范围 + 建议的取舍,等定夺 |
| 3 设计 | 审查技术 / 体验方案,挑选权衡 | 给出方案和取舍,**不替开发者做不可逆决策** |
| 4-5 搭骨架 / 迭代 | 给业务方向、回答求助、决定何时发布 | 自动推进 + 自动提交,关键节点停下汇报 |
| 6-7 验证 / 发布 | 验收、决定上线时机 | 报告真实结果(失败就说失败),不替开发者点"发布" |

### 把决策权和执行权分开

- **可逆的执行**(写代码、跑测试、本地提交)→ AI 自动做,事后能 review。
- **不可逆的动作**(`git push`、部署、删数据、对外发请求)→ **永远手动确认**,AI 不擅自触发。

### 卡住不死磕

- 同一个点连续卡几次(比如 3 次)就**升级成"求助"**:停下来,讲清楚卡在哪、试过什么、需要开发者什么决定,而不是反复试错烧时间/token。

### 状态可接力

- 进度、决策、当前在做哪一步都落到文档里(而不是只存在某次对话里),换个新会话或新人也能读几个文件就跟上,不用从头复述。

> 一句话:**AI 负责把活干快干对,开发者负责掌舵和按下不可逆的按钮。** 边界越清楚,协作越顺。
